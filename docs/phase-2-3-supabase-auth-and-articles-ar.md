# المرحلتان الثانية والثالثة: ربط تسجيل الدخول وحفظ المقالات مع Supabase

تم تنفيذ الربط داخل الكود. هذا الدليل يشرح لك ما يجب فعله حتى يعمل التطبيق عندك.

تم تجاهل n8n بالكامل في هذه المرحلة كما طلبت.

## ماذا تغيّر في التطبيق؟

### تسجيل الدخول

كان تسجيل الدخول سابقًا يعتمد على أسماء وكلمات مرور مكتوبة داخل الكود.

الآن أصبح تسجيل الدخول عبر Supabase Auth:

- المستخدم يدخل بالبريد الإلكتروني.
- كلمة المرور تكون من Supabase.
- لا نستخدم قائمة `USERS` القديمة للدخول.

### حفظ المقالات

كان حفظ المقالات سابقًا داخل جهاز المستخدم فقط عبر `localStorage`.

الآن الحفظ الأساسي أصبح في جدول:

`public.articles`

داخل Supabase.

المسودات المحلية ما زالت موجودة فقط كنسخة إنقاذ على نفس الجهاز، لكنها ليست مصدر البيانات الرئيسي.

## الخطوة 1: أحضر بيانات Supabase

افتح Supabase ثم:

1. ادخل إلى مشروعك.
2. من القائمة الجانبية اختر `Project Settings`.
3. اختر `API`.
4. ستجد:
   - `Project URL`
   - `anon public key`

مهم جدًا:

استخدم `anon public key` فقط داخل تطبيق الواجهة.

لا تستخدم `service_role key` داخل `.env.local` ولا داخل أي كود يظهر في المتصفح.

## الخطوة 2: عدّل ملف `.env.local`

افتح ملف:

`.env.local`

وأضف هذه القيم:

```env
VITE_SUPABASE_URL=https://your-project-ref.supabase.co
VITE_SUPABASE_ANON_KEY=your-supabase-anon-public-key
```

استبدل القيم ببيانات مشروعك من Supabase.

مثال شكلي فقط:

```env
VITE_SUPABASE_URL=https://abcdefghijk.supabase.co
VITE_SUPABASE_ANON_KEY=eyJhbGciOi...
```

لا تحذف مفاتيح Gemini الآن إذا كنت تحتاجها محليًا، لكن لاحقًا عند النشر سننقل مفاتيح الذكاء الاصطناعي إلى السيرفر.

## الخطوة 3: تأكد من وجود المستخدمين في Supabase

افتح:

`Authentication > Users`

أنشئ مستخدمًا أو أكثر.

كل مستخدم يجب أن يدخل إلى المحرر باستخدام:

- البريد الإلكتروني
- كلمة المرور

ولجعل حسابك Admin، استخدم الأمر الذي طبقناه في دليل المرحلة الأولى:

```sql
update public.profiles
set role = 'admin'
where email = 'YOUR_EMAIL@example.com';
```

## الخطوة 4: شغّل التطبيق محليًا

في مجلد المشروع شغّل:

```bash
npm install
npm run dev
```

افتح الرابط الذي يظهر لك في الطرفية، غالبًا:

```text
http://localhost:3000
```

أو:

```text
http://localhost:5173
```

حسب المنفذ المستخدم.

## الخطوة 5: اختبر تسجيل الدخول

في شاشة الدخول:

1. اكتب البريد الإلكتروني الموجود في Supabase.
2. اكتب كلمة المرور.
3. اضغط تسجيل الدخول.

إذا فشل الدخول:

- تأكد أن البريد وكلمة المرور صحيحان.
- تأكد أنك وضعت `VITE_SUPABASE_URL`.
- تأكد أنك وضعت `VITE_SUPABASE_ANON_KEY`.
- أعد تشغيل `npm run dev` بعد تعديل `.env.local`.

## الخطوة 6: اختبر حفظ المقالات

بعد الدخول:

1. اضغط إنشاء مقال جديد.
2. اكتب عنوانًا.
3. اكتب نصًا داخل المحرر.
4. اضغط حفظ المسودة اليدوية.
5. افتح Supabase.
6. ادخل إلى `Table Editor`.
7. افتح جدول `articles`.

يجب أن ترى المقال محفوظًا هناك.

ستجد بيانات مثل:

- `title`
- `content_json`
- `content_html`
- `plain_text`
- `keywords`
- `goal_context`
- `analysis`
- `stats`
- `owner_id`
- `last_saved_at`

## الخطوة 7: اختبر لوحة التحكم

بعد حفظ مقال:

1. ارجع إلى لوحة التحكم.
2. اضغط تحديث إذا لم يظهر المقال فورًا.
3. يجب أن يظهر المقال من Supabase.
4. اضغط عليه لفتحه في المحرر.
5. عدّل النص واحفظ.
6. تأكد أن نفس الصف في Supabase يتحدث، ولا يتم إنشاء صف جديد كل مرة.

## ماذا يرى المستخدم العادي؟

المستخدم العادي يرى:

- مقالاته الخاصة.
- أي مقالات مشتركة إذا أضفنا لاحقًا `visibility = shared`.

حاليًا المقالات التي ينشئها المستخدم تحفظ كـ:

```text
visibility = private
source = manual
status = draft
```

## ماذا يرى الأدمن؟

إذا كان حسابك `admin` في جدول `profiles`، فسيستطيع قراءة كل المقالات حسب سياسات RLS التي أنشأناها في المرحلة الأولى.

لوحة أدمن تفصيلية باسم كل مستخدم لم ننفذها بعد، لأنها مرحلة لاحقة، لكن قاعدة البيانات والصلاحيات جاهزة لها.

## ملاحظة مهمة عن المقالات القديمة

المقالات القديمة التي كانت محفوظة فقط داخل `localStorage` على جهازك لن تظهر في لوحة Supabase الجديدة تلقائيًا.

هي لم تُحذف من جهازك، لكنها ليست داخل قاعدة البيانات.

إذا أردت نقل المقالات القديمة إلى Supabase، أخبرني وسأنفذ لك أداة ترحيل داخل التطبيق أو سكربت مخصص.

## مشاكل شائعة وحلولها

### تظهر رسالة خطأ عند تسجيل الدخول

الحلول:

1. تأكد من البريد وكلمة المرور.
2. تأكد أن المستخدم موجود في `Authentication > Users`.
3. تأكد من وجود سجل له في `profiles`.
4. أعد تشغيل التطبيق بعد تعديل `.env.local`.

### المقال لا يظهر في لوحة التحكم

الحلول:

1. اضغط زر التحديث.
2. تأكد أنك ضغطت حفظ داخل المحرر.
3. افتح جدول `articles` في Supabase وتأكد أن الصف موجود.
4. إذا ظهر الصف في Supabase ولم يظهر في التطبيق، أرسل لي رسالة الخطأ من Console.

### يظهر خطأ RLS أو permission denied

هذا يعني غالبًا أن سياسات Supabase لم تطبق كما يجب.

راجع المرحلة الأولى وشغّل ملف SQL مرة أخرى، أو أرسل لي نص الخطأ.
